## 2. 보안 고려사항 (Security Considerations)
싱글 플레이 환경을 목표로 하더라도, 클라이언트와 서버를 분리한 구조이므로 기본적인 웹 서버 보안 원칙을 준수해야 합니다.

### A. 클라이언트 신뢰 금지 (No Client Trust)
* **핵심 비즈니스 로직 캡슐화:** 유니티(C#) 코드 내에서 믹스 확률을 계산하거나 전투 승패를 결정하면 안 됩니다. 클라이언트는 단지 "A와 B 헨치를 믹스하겠다"는 **요청(Request)**만 서버로 전송해야 합니다.
* **서버 검증:** 실제 난수 생성(확률 계산), 마스터리 포인트 누적, 인벤토리 아이템 차감/지급 등은 모두 Tomcat 서버(Java 로직)에서 처리한 뒤, 그 **결과값(Response)**만 클라이언트로 반환해야 무결성이 유지됩니다.

### B. 데이터베이스 보안 (SQL Injection 방지)
* JSP나 Java 코드로 MySQL과 통신할 때, 클라이언트에서 넘어온 값(예: 닉네임, 검색어 등)을 SQL 쿼리문에 직접 문자열 연산(String Concat)으로 이어 붙이지 마세요.
* 반드시 `PreparedStatement`를 사용하여 파라미터를 바인딩함으로써 SQL 인젝션 공격을 원천 차단해야 합니다.

### C. 민감 정보 하드코딩 금지
* `server/` 코드 내에 MySQL 접속 계정이나 비밀번호(`root` 등)를 평문으로 적어두고 GitHub에 그대로 푸시(Push)하는 것은 위험합니다.
* DB 연결 정보는 외부 설정 파일(`properties` 또는 `.env`)로 분리하고, 해당 설정 파일은 반드시 `.gitignore`에 등록하여 레포지토리에 노출되지 않도록 합니다.

### D. 통신 구간 보안 및 패킷 변조 방지 (Network Security)
* **HTTPS/TLS(SSL) 통신:** 로컬 테스트 단계를 벗어나 외부 서버에 배포할 경우, 클라이언트와 서버 간의 API 통신은 반드시 HTTPS 프로토콜을 적용하여 데이터 스니핑(가로채기)을 원천 차단해야 합니다.
* **패킷 변조 검증:** 악의적인 유저가 중간에 데이터를 조작(예: 믹스 재료 ID를 임의의 희귀 헨치로 변경)하는 것을 막기 위해, 중요 요청 시 클라이언트 측에서 파라미터 값들에 대한 해시(Hash)나 체크섬(Checksum)을 생성하여 함께 서버로 보내고, 서버에서 이를 대조하는 검증 로직 도입을 고려해야 합니다.
